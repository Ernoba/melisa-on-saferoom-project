Core Concepts
MELISA is built around three simple ideas: containers, users, and projects.

📦 Containers
Under the hood, MELISA uses LXC (Linux Containers). Each container is a lightweight,
isolated environment that shares the host’s kernel but has its own filesystem, network,
and process space. You can create containers from any distribution that LXC supports
(see melisa --search).
MELISA adds a few convenience layers on top of plain LXC:

Automatic network configuration – containers get a veth pair connected to lxcbr0.
DNS setup – Google’s 8.8.8.8 is injected into /etc/resolv.conf.
Initial package manager sync – apt update, dnf makecache, etc., run automatically.
Easy folder sharing – mount host directories into a container with melisa --share.
Graceful deletion – stops the container if running, then destroys it.
In the source code, container operations are handled in src/core/container.rs.
The create_new_container function uses tokio::process::Command to call lxc-create,
then injects network config, sets up DNS, starts the container, and runs a distro‑specific
update command (determined by the DistroMetadata.pkg_manager field).
👥 Users
MELISA introduces two levels of users on the host:

Regular users – can only manage containers (create, start, stop, delete their own).
Admins – can also manage users (--add, --remove, --upgrade) and projects.
All MELISA users have their shell set to /usr/local/bin/melisa – that’s the jail shell.
When they log in via SSH, they land directly in the MELISA prompt, not a full system shell.
This is enforced by sudoers rules and file permissions.
When you add a user with melisa --add, you’re prompted to choose a role. The system then:

Creates a system user with home directory (/home/username).
Sets the shell to the MELISA binary.
Sets permissions on the home directory to 700 (private).
Prompts you to set a password.
Writes a sudoers file (/etc/sudoers.d/melisa_username) with appropriate commands.
The user management code lives in src/core/user_management.rs. It uses tokio::process::Command
to call system utilities (useradd, passwd, userdel, etc.) and writes sudoers rules via a
tee subprocess.
📁 Projects
Projects are Git repositories that live in /opt/melisa/projects (the master copies).
When an admin invites a user to a project, the user gets a clone of that repository
in their home directory (/home/username/projectname). They can work there, commit, and
push changes. The admin can later --pull changes from a user’s branch back into the master.
This turns MELISA into a simple collaboration platform: everyone works in their own sandboxed
environment, but code flows through Git.
Project management functions are implemented in src/core/project_management.rs.
Key functions:
new_project – creates a bare Git repo in /opt/melisa/projects.
invite – clones the master repo to each invited user’s home.
pull – adds, commits, and pushes changes from a user’s folder to the master.
delete_project – removes master and all user copies.
out_user – removes a project from a specific user.
🌐 The Big Picture

Security & Permissions
MELISA takes security seriously. Here’s how we keep things locked down.

🔒 Jail Shell
When you add a user with melisa --add, their login shell is set to /usr/local/bin/melisa.
That’s not a real shell – it’s the MELISA binary itself. When they SSH into the host,
they’re greeted by the MELISA prompt, not a bash prompt. They can only run melisa commands,
not arbitrary system commands.
This is enforced by:

The shell being listed in /etc/shells
Sudoers rules that grant passwordless melisa execution
Home directories with chmod 700 so others can’t peek
🛡️ Sudoers Rules
Admins get additional sudoers entries that allow them to run user‑management tools (useradd,
userdel, passwd, etc.) without a password. Regular users only get LXC commands.
All sudoers files are placed in /etc/sudoers.d/melisa_username and have strict permissions 0440.
The configure_sudoers function in user_management.rs builds a list of allowed commands
based on the user’s role, then writes the file using sudo tee.
🧹 Orphan Cleanup
If you delete a user with --remove, their sudoers file is automatically removed.
The --clean command scans for leftover melisa_* files in /etc/sudoers.d and deletes them
if the corresponding user no longer exists.
🚫 No Remote Setup
The melisa --setup command checks if you’re in an SSH session and refuses to run.
This prevents a remote attacker who somehow gained access from reconfiguring the host.
rust

// From root_check.rspub async fn is_ssh_session() -> bool {    if !env::var("SSH_CLIENT").unwrap_or_default().is_empty() { return true; }    // ... also checks `who -m`}
🧊 Filesystem Privacy
During setup, MELISA runs chmod 711 /home. That means users can still cd into their own
home, but they cannot list other users’ directories (ls /home returns nothing).
This is a simple but effective privacy measure.
🌐 Network Hardening
Containers are connected to lxcbr0, a NATed bridge.
The host firewall (firewalld) is configured to trust lxcbr0.
SSH port 22 is opened, but only for authorized users.
Subuid/subgid mappings are set up for each user (/etc/subuid, /etc/subgid).
🔑 SSH Key Enforcement
When you use the client, it automatically generates an SSH key (if missing) and copies it to
the server with ssh-copy-id. After that, no passwords are used – only keys.
Multiplexing is enabled to speed up repeated connections.
📁 Shared Folders & Permissions
When you share a host folder with a container, MELISA writes an lxc.mount.entry into the
container’s config. The folder is bind‑mounted, and inside the container it appears with the
mapped UID/GID (usually starting at 100000). This prevents container processes from escaping
to the host.
🧪 Summary
MELISA applies the principle of least privilege everywhere:

Users can only run what they need.
The setup is irreversible from remote.
The filesystem is locked down.
Containers are isolated.
All this makes MELISA a safe playground for experiments and team projects.

Troubleshooting
Even the smoothest tools hit a bump sometimes. Here are common issues and how to fix them.

❌ Permission denied when running melisa
Symptom: error: Permission not allowed. This function is for admin only.
Why: You’re trying to run an admin command as a regular user, or your sudoers file is missing.
Fix:

If you’re the admin, make sure you were added with the Admin role. Check with melisa --users.
If you’re a regular user, you can still use container commands (--create, --run, etc.).
If sudoers is broken, manually inspect /etc/sudoers.d/melisa_* – they should contain the right commands.
🌐 --search shows no results or old cache
Symptom: The list of distributions is empty or outdated.
Why: MELISA caches the LXC image list in /tmp/melisa_global_distros.cache for one hour.
If the cache is stale and the network is down, you might see nothing.
Fix:

Wait for the cache to expire, or delete it manually: sudo rm /tmp/melisa_global_distros.cache
Ensure your host has internet access and can reach the LXC image servers.
🐢 Container creation hangs or fails with GPG errors
Symptom: [ERROR] Masalah tanda tangan GPG. or Gagal mendownload template.
Why: LXC templates need to verify GPG signatures, and sometimes keys are missing on the host.
Fix:

Run gpg --recv-keys <keyid> (the error message usually shows the missing key).
Or try a different distribution/image.
🔄 After sharing a folder, it’s not visible in the container
Symptom: You added a shared folder, but inside the container the directory is empty.
Why: The container must be restarted for mount changes to take effect.
Fix:
bash

melisa --stop mycontainermelisa --run mycontainer
👤 User cannot log in (password rejected)
Symptom: SSH login fails with “Permission denied (publickey,password)”.
Why: The user’s password might not be set, or they don’t have a valid shell.
Fix:

As admin, run melisa --passwd username to set/reset the password.
Check /etc/passwd – the shell should be /usr/local/bin/melisa.
📁 Project cloning fails with “does not appear to be a git repository”
Symptom: When inviting a user, you see fatal: ... does not appear to be a git repository.
Why: The master project directory was not initialised as a bare Git repo.
Fix:

Re‑create the project with melisa --new_project <name>.
Or manually: cd /opt/melisa/projects/<name> && git init --bare.
🧹 Orphaned sudoers files not cleaned
Symptom: melisa --users shows trash files but --clean didn’t remove them.
Why: The clean command checks existing users from /etc/passwd. If a user was deleted manually,
the file may linger.
Fix: Run sudo rm /etc/sudoers.d/melisa_<username> manually, then melisa --clean again.

🧪 Still stuck?
Open an issue on the GitHub repository with the exact
error message and steps to reproduce. We’re happy to help!

Source Code Deep Dive
This chapter explores the internals of MELISA, explaining how each module works and why certain design decisions were made.

🦀 Rust Core
The Rust code is structured into three main modules: cli, core, and distros.

src/core/ – The Heart of MELISA
container.rs
This file contains all LXC container operations. Key functions:

create_new_container(name, meta) – calls lxc-create, injects network config, sets DNS, starts the container, and runs an initial package manager update.
delete_container(name) – stops the container if running, then destroys it.
add_shared_folder / remove_shared_folder – edits the container’s config file to add/remove bind mounts.
send_command – checks if the container is running, then executes a command inside it using lxc-attach.
upload_to_container – accepts a tar stream on stdin and extracts it inside the container.
All functions are async and use tokio::process::Command to avoid blocking the main thread.

user_management.rs
Handles creation, deletion, and permission management of MELISA users.

add_melisa_user – creates a system user with the MELISA shell, sets home permissions, prompts for password, and calls configure_sudoers.
configure_sudoers – writes a sudoers file with a list of allowed commands based on the user’s role (Admin or Regular). It uses sudo tee to write the file.
list_melisa_users – parses /etc/passwd for users with the MELISA shell, and also scans /etc/sudoers.d for orphaned files.
clean_orphaned_sudoers – removes sudoers files whose corresponding users no longer exist.
project_management.rs
Implements the Git-based collaboration features.

new_project – runs git init --bare in /opt/melisa/projects/<name>.
invite – clones the master repo to each user’s home and fixes ownership.
pull – performs git add ., git commit, and git push inside the user’s working copy.
delete_project – removes the master and all user copies (by scanning /etc/passwd).
update_project – does a normal git pull or a forced reset --hard.
All Git commands are spawned as child processes with tokio::process::Command.

root_check.rs
Utility functions for privilege and session checks:

check_root() – uses libc::geteuid() to see if running as root.
check_if_admin(username) – checks if the user has a sudoers file that includes admin‑level commands (like useradd). It uses sudo -n to avoid password prompts.
is_ssh_session() – inspects environment variables and the output of who -m to detect SSH sessions.
setup.rs
The install() function performs the one‑time host setup. It:

Verifies it’s run as root and not over SSH.
Executes a series of system commands (package installs, service enables, etc.) with progress feedback.
Deploys the MELISA binary with SUID.
Sets up firewall rules, subuid mappings, and the projects directory.
Registers the shell and writes the global sudoers rule.
src/cli/ – User Interface
executor.rs
The execute_command function is the main dispatcher for both built‑in commands and external programs. It:

Matches on melisa subcommands and calls the appropriate async functions.
For cd, it changes the current directory using std::env::set_current_dir.
For any other command, it spawns a bash process with a custom PATH (including ~/.cargo/bin) and appropriate environment variables (HOME, USER, RUSTUP_HOME, etc.). This allows users to run Rust tools like cargo inside the jail.
The function returns an ExecResult enum (Continue, Break, ResetHistory, Error) which the main loop uses to decide whether to save history, exit, or clear history.

prompt.rs
The Prompt struct builds the interactive prompt (e.g., melisa@user:~/project>). It reads the current user and home directory from environment variables, and replaces the home path with ~ for brevity.
The reset_history function clears the in‑memory history and deletes the history file. It’s called when --clear is issued.

helper.rs
Implements a custom rustyline helper for line editing, history, and tab completion. It:

Provides filename completion for cd and paths.
Searches the command history for suggestions that match the current input prefix.
Highlights matching brackets and hints.
loading.rs
A simple spinner utility using the indicatif crate. It runs a future while displaying a spinner with an elapsed timer.

wellcome.rs
Displays the fancy boot banner with system information, GPU detection, and a glitch‑style animation. It uses colored, rand, sysinfo, and chrono to create a professional dashboard.

src/distros/ – LXC Image List Handling
distro.rs
The get_lxc_distro_list function fetches the list of available LXC images by invoking lxc-download --list. It implements a global cache with a lock file to avoid multiple concurrent updates. The cache is stored in /tmp/melisa_global_distros.cache and expires after one hour.
If the network is unavailable, it falls back to the cached list.
The raw output is parsed into a vector of DistroMetadata structs, each containing:

slug: a unique identifier (e.g., ubu-22.04-x64)
name, release, arch, variant
pkg_manager: inferred from the distribution name (apt, dnf, apk, pacman, etc.)
This metadata is used during container creation to run the correct package manager update.

🐚 Client Scripts
The client is written in Bash for maximum portability. It resides in src/melisa_client/.

auth.sh
Manages server profiles:

auth_add – prompts for a name and user@host, copies SSH key, sets up multiplexing, and saves the profile.
auth_switch – writes the active profile name to ~/.config/melisa/active.
auth_list – displays all profiles, marking the active one.
auth_remove – deletes a profile and clears the active file if needed.
exec.sh
Handles remote command execution:

exec_run – uploads a file and runs it non‑interactively.
exec_run_tty – uploads, runs with a pseudo‑terminal, then cleans up.
exec_upload – tars a directory and streams it to melisa --upload on the server.
exec_forward – simply forwards any unknown command via SSH.
utils.sh
Provides common utilities:

Colour definitions and logging functions (log_info, log_success, log_error).
ensure_ssh_key – generates an RSA key if missing.
install.sh
Copies the scripts to ~/.local/share/melisa/, creates the main melisa launcher in ~/.local/bin/, and updates PATH in ~/.bashrc.

🔄 Async Design with Tokio
MELISA is built on Tokio, Rust’s asynchronous runtime. This allows it to perform many I/O operations concurrently without blocking. For example:

While waiting for lxc-create to finish, the main thread can still respond to other requests.
Multiple containers can be managed simultaneously.
The spinner animation runs while the underlying command executes.
All functions that call external commands are marked async and use .await to yield control back to the runtime.

🧠 Lessons Learned
Locking for cache updates:
 The global distro list uses a lock file to prevent multiple processes 
from updating the cache at the same time. This avoids corrupted cache 
files.
Graceful error handling: MELISA checks for common errors (e.g., container already exists, GPG key missing) and provides helpful messages.
Security by design: The jail shell, sudoers restrictions, and SSH session checks are all implemented from the ground up.
Now that you’ve seen the internals, you’re ready to contribute or even extend MELISA for your own needs!
