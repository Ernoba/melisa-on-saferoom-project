Identitas, Filosofi & Visi MELISA

MELISA adalah akronim dari Management Environment Linux Sandbox — sebuah sistem orkestrasi container berbasis LXC yang ditulis sepenuhnya dalam bahasa pemrograman Rust. Dikembangkan oleh Erick Adriano Sebastian (ernobaproject@gmail.com) di bawah proyek yang bernama Saferoom, MELISA lahir dari sebuah frustrasi yang sangat nyata di dunia pengembangan perangkat lunak: masalah Host Pollution.

Host Pollution adalah kondisi di mana sistem operasi utama seorang developer semakin kotor dan tidak stabil karena berisi tumpukan dependensi dari berbagai proyek yang berbeda — Python 3.8 dari proyek A bertabrakan dengan Python 3.11 dari proyek B, paket npm global yang saling bentrok, variabel environment yang bocor, dan library sistem yang terkontaminasi. MELISA hadir dengan satu tujuan mutlak: setiap pekerjaan dilakukan di dalam ruang bersih yang terisolasi, dan ketika selesai, ruang itu bisa dibuang atau dipertahankan tanpa pernah mengotori host.

Filosofi "Saferoom" dan "Clean Room"

Nama proyek Saferoom bukan sekadar nama keren. Ia mencerminkan filosofi inti: setiap container MELISA adalah ruang aman di mana developer bisa bereksperimen, merusak, dan membangun ulang tanpa konsekuensi pada sistem utama. Pengembang menggunakan analogi "butler" — MELISA seperti pelayan yang selalu menyiapkan clean room setiap kali Anda ingin coding, dan membersihkannya kembali saat Anda selesai.

🦀

Mengapa Rust?

Rust dipilih bukan karena tren, melainkan karena kebutuhan. Memory safety tanpa garbage collector berarti MELISA tidak akan pernah crash akibat dangling pointer atau null dereference — kritis untuk sebuah tool yang mengelola container sistem. Tokio async runtime memberikan concurrency setara Go namun dengan overhead jauh lebih kecil. Binary Rust yang dioptimasi bisa berukuran sangat kecil dan tidak memerlukan runtime tambahan di server.

🐧

Mengapa LXC?

LXC (Linux Containers) adalah teknologi virtualisasi tingkat OS yang sudah tertanam langsung di kernel Linux sejak versi 2.6.24. Berbeda dari Docker yang membangun abstraksi di atas namespaces dan cgroups, LXC menggunakan mekanisme tersebut secara langsung — menghasilkan container yang lebih "mirip VM" (punya init process sendiri, systemd, service daemon) namun dengan overhead yang mendekati nol.

🔗

Hybrid Architecture

Keputusan untuk menulis server dalam Rust dan klien dalam Bash adalah keputusan brilian. Server memerlukan performa, safety, dan akses ke API sistem — Rust ideal. Klien cukup menjadi relay cerdas via SSH — Bash sempurna karena tersedia di hampir setiap mesin Linux/macOS tanpa instalasi apapun.

👥

Dirancang untuk Tim

Tidak seperti tool container lain yang berfokus pada deployment aplikasi, MELISA dirancang dari awal untuk skenario multi-user — tim developer yang berbagi satu server, dengan proyek bersama, kontrol akses berbeda per orang, dan alur kolaborasi yang terintegrasi langsung ke dalam tool tanpa membutuhkan platform eksternal.

🏰

Security-First Design

Di MELISA, keamanan bukan fitur tambahan yang ditempel setelah sistem dibangun. Ia adalah bagian dari setiap keputusan desain sejak awal — dari mengapa setup wajib fisik, hingga mengapa setiap user punya file sudoers tersendiri, hingga mengapa DNS di dalam container dikunci dengan atribut immutable.

📖

Open Source & MIT

Dilisensikan di bawah MIT License, MELISA sepenuhnya terbuka untuk digunakan, dimodifikasi, dan didistribusikan — termasuk untuk keperluan komersial. Ini mencerminkan komitmen pengembang terhadap komunitas open-source Linux.

⚡ Dampak Release Optimization

Kombinasi opt-level='z' + lto=true + strip=true menghasilkan binary MELISA yang sangat kecil (estimasi <5MB) tanpa overhead runtime. Ini berarti MELISA bisa dideploy bahkan di server dengan storage terbatas, dan waktu startup hampir instan. panic='abort' juga berarti tidak ada kemungkinan stack unwinding yang bisa dieksploitasi untuk information leakage.

Dukungan Multi-Distro Linux

Salah satu aspek MELISA yang paling kuat dan sering disalahpahami adalah kemampuan multi-distribusi Linux-nya. Berbeda dengan banyak dokumentasi awal yang menyebut hanya Fedora, analisis mendalam terhadap modul src/distros/host_distro.rs mengungkapkan bahwa MELISA telah membangun infrastruktur deteksi dan adaptasi distro yang cukup lengkap — mencakup seluruh keluarga besar Linux.

Mekanisme Deteksi Host OS

MELISA mendeteksi sistem operasi host secara runtime dengan membaca file standar /etc/os-release secara asinkron menggunakan tokio::fs. Pendekatan ini adalah cara yang benar dan paling kompatibel karena file /etc/os-release adalah standar FreeDesktop.org yang diimplementasikan di hampir seluruh distro Linux modern.

// host_distro.rs — Deteksi OS via /etc/os-release pub async fn detect_host_distro() -> HostDistro { let content = fs::read_to_string("/etc/os-release") .await .unwrap_or_default() .to_lowercase(); // Case-insensitive matching if content.contains("fedora") || content.contains("rhel") || content.contains("centos") || content.contains("rocky") { HostDistro::Fedora // RHEL family: dnf + firewalld } else if content.contains("ubuntu") { HostDistro::Ubuntu // Ubuntu: apt-get + ufw } else if content.contains("debian") { HostDistro::Debian // Debian: apt-get + ufw } else if content.contains("arch") { HostDistro::Arch // Arch Linux: pacman + iptables } else { // Fallback: Ekstrak ID dari os-release untuk logging HostDistro::Unknown(id) // Default ke apt-get/ufw/ssh } }

Tiga Jenis Firewall yang Didukung

Salah satu detail teknis yang sangat penting adalah bahwa MELISA tidak hanya mendukung satu sistem firewall — ia mendukung tiga yang paling umum digunakan di Linux, dan memilih secara otomatis berdasarkan distro yang terdeteksi:

// Firewall abstraction — host_distro.rs pub enum FirewallKind { Firewalld, // RHEL/Fedora — menggunakan zone "trusted" untuk lxcbr0 Ufw, // Ubuntu/Debian — mengizinkan traffic masuk di lxcbr0 Iptables, // Arch/minimal — menambahkan rule ACCEPT di chain INPUT } // Contoh eksekusi untuk Firewalld: firewall-cmd --zone=trusted --add-interface=lxcbr0 --permanent firewall-cmd --reload // Contoh eksekusi untuk UFW: ufw allow in on lxcbr0 ufw reload // Contoh eksekusi untuk iptables: iptables -I INPUT -i lxcbr0 -j ACCEPT

Package Manager Kontainer (di dalam Container)

Selain package manager host, MELISA juga mengenal package manager dari berbagai distribusi yang bisa berjalan di dalam container. Saat container dibuat, MELISA secara otomatis menjalankan update awal menggunakan package manager yang tepat:

🌐 Distro Container vs Distro Host — Bebas Kombinasi

Ini adalah keunggulan penting: distro host dan distro container bisa berbeda sepenuhnya. Anda bisa menjalankan MELISA di server Fedora, namun membuat container Ubuntu, Alpine, Arch, atau bahkan Kali Linux di dalamnya — semua berjalan secara bersamaan. Inilah mengapa MELISA sangat fleksibel untuk testing lintas-distro.

Langkah 3 — Menjalankan --setup

# Di dalam REPL MELISA, jalankan: melisa@saferoom:../saferoom> melisa --setup

Perintah ini menjalankan wizard inisialisasi lengkap. Berikut detail setiap sub-operasi yang dilakukan:

1
System Update

Menjalankan update package manager (dnf update -y / apt-get update / pacman -Sy) untuk memastikan semua paket terbaru tersedia.

2
Instalasi Dependensi LXC

Menginstal paket lxc, lxc-templates (atau lxc-utils), libvirt, bridge-utils, paket SSH server, dan firewalld / ufw sesuai distro.

3
Konfigurasi Jaringan

Memuat kernel module veth, mengaktifkan dan memulai lxc-net.service yang membuat bridge lxcbr0 (biasanya di subnet 10.0.3.0/24).

4
Konfigurasi Firewall

Mengotorisasi port SSH (22/tcp) dan mengizinkan seluruh traffic melalui interface lxcbr0 di zone trusted (firewalld), atau via allow rule di UFW/iptables.

5
Deploy Binary MELISA

Menyalin binary hasil kompilasi ke /usr/local/bin/melisa dan menerapkan SUID bit (chmod 4755). Ini memungkinkan pengguna non-root menjalankan MELISA dengan eskalasi privilege terkontrol.

6
Register Jail Shell

Mendaftarkan /usr/local/bin/melisa ke /etc/shells. Ini diperlukan agar sistem mengizinkan penggunaan MELISA sebagai login shell pengguna.

7
Infrastruktur Proyek

Membuat direktori /opt/melisa/projects dengan Sticky Bit 1777 — sehingga semua pengguna bisa membuat direktori di dalamnya, tapi hanya pemilik yang bisa menghapus direktori miliknya.

8
User Namespace Mapping

Mengonfigurasi subuid dan subgid range (100000-165535) untuk pengguna yang sedang melakukan setup. Ini adalah prasyarat untuk container yang berjalan dalam unprivileged mode.

9
Pengerasan Keamanan

Menerapkan chmod 711 /home untuk mencegah enumerasi pengguna, mengkonfigurasi Git global safe.directory, dan memperbaiki permissions pada newuidmap/newgidmap.

10
Sudoers Global

Membuat /etc/sudoers.d/melisa dengan rule ALL ALL=(ALL) NOPASSWD: /usr/local/bin/melisa bererizin 0440, memungkinkan semua pengguna menjalankan MELISA tanpa password sudo.

Langkah 4 — Verifikasi

# Setelah setup, verifikasi dengan: melisa@saferoom:../saferoom> melisa --list # Output: Daftar kosong (normal — belum ada container) # Jika tidak ada error, setup berhasil!

Panduan Instalasi Klien (Remote) — Step by Step

Klien MELISA adalah script Bash ringan yang bisa diinstal di laptop, desktop, atau workstation manapun yang memiliki SSH client. Tidak ada dependensi binary tambahan selain Bash dan SSH standar.

Langkah 1 — Menjalankan Installer

# Masuk ke direktori client dalam repository cd melisa-on-saferoom-project/src/melisa_client # Jalankan installer ./install.sh

Installer akan melakukan hal-hal berikut secara otomatis:

  • Membuat ~/.local/bin/ dan ~/.local/share/melisa/
  • Menyalin binary melisa ke ~/.local/bin/melisa
  • Menyalin semua modul (auth.sh, exec.sh, utils.sh) ke ~/.local/share/melisa/
  • Menambahkan ~/.local/bin ke PATH di ~/.bashrc, ~/.zshrc, atau ~/.profile sesuai shell aktif
  • Memperbaiki permission ownership jika sebelumnya direktori dikunci oleh root
# Setelah instalasi, reload shell source ~/.bashrc # atau source ~/.zshrc # Verifikasi instalasi melisa # Akan menampilkan help menu jika berhasil

Langkah 2 — Menambahkan Profil Server

# Format: melisa auth add [nama-profil] [user@ip-server] melisa auth add myserver root@192.168.1.100 # Apa yang terjadi di balik layar: # 1. Cek SSH key di ~/.ssh/id_ed25519 atau ~/.ssh/id_rsa # 2. Jika tidak ada → buat Ed25519 keypair otomatis # 3. ssh-copy-id root@192.168.1.100 (prompt password sekali) # 4. Tambahkan SSH ControlMaster config ke ~/.ssh/config # 5. Simpan profil di ~/.config/melisa/profiles.conf # 6. Set profil ini sebagai active

Konfigurasi SSH ControlMaster (Diinjeksi Otomatis)

Ini adalah detail teknis yang penting — MELISA secara otomatis mengkonfigurasi SSH multiplexing di ~/.ssh/config:

# ~/.ssh/config — diinjeksi oleh melisa auth add Host 192.168.1.100 User root ControlMaster auto ControlPath ~/.ssh/sockets/%r@%h:%p ControlPersist 10m

ControlMaster berarti SSH membuka satu koneksi master dan semua perintah selanjutnya menggunakan socket yang sama — menghemat waktu handshake SSL yang biasanya 200-500ms per perintah. Dengan ControlPersist 10 menit, koneksi tetap hidup selama 10 menit setelah perintah terakhir.

Manajemen Multi-Server

# Tambah server kedua melisa auth add prodserver admin@203.0.113.50 # Lihat semua server terdaftar melisa auth list # Output: # * myserver (root@192.168.1.100) <- [ACTIVE] # prodserver (admin@203.0.113.50) # Ganti server aktif melisa auth switch prodserver # Hapus profil melisa auth remove myserver

Sistem Manajemen Container — Analisis Teknikal Mendalam

LXC: Pilihan Teknologi yang Tepat

Memilih LXC atas Docker adalah keputusan arsitektur yang sangat disengaja. LXC (Linux Containers) menggunakan kernel namespaces dan cgroups secara langsung tanpa layer daemon tambahan. Hasilnya adalah container yang lebih "transparan" — developer bisa melihat proses container di host dengan ps aux, resource usage langsung terbaca tanpa overhead, dan ada tidak ada "magic layer" yang menyembunyikan perilaku sistem.

Alur Pembuatan Container (--create)

// container.rs — create_container() dipanggil setelah user memilih distro dari TUI pub async fn create_container(name: &str, distro: &DistroMetadata) { // 1. Pastikan host network siap (lxcbr0 bridge aktif) ensure_host_network_ready().await; // 2. Eksekusi lxc-create via download template // lxc-create -P /var/lib/lxc -n [name] -t download // -- --dist [distro] --release [release] --arch [arch] run_lxc_create(name, distro).await; // 3. Injeksi konfigurasi jaringan ke config file container inject_network_config(name).await; // 4. Setup DNS statik dan kunci dengan chattr +i setup_container_dns(name).await; // 5. Auto-update package repo sesuai pkg_manager distro auto_initial_setup(name, &distro.pkg_manager).await; // 6. Tulis metadata MELISA ke /etc/melisa-info dalam container write_container_metadata(name, distro).await; }

Random MAC Address Generation

Detail menarik: setiap container mendapat MAC address yang di-generate secara random dengan prefix tetap ee:ec:fa:5e:. Dua byte terakhir adalah random:

// inject_network_config() — MAC address randomization let net_config = format!( "lxc.net.0.hwaddr = ee:ec:fa:5e:{:02x}:{:02x}\n", rand::random::<u8>(), // byte ke-5: 0x00-0xFF random rand::random::<u8>() // byte ke-6: 0x00-0xFF random );

Randomisasi MAC address adalah praktik keamanan penting karena mencegah fingerprinting container berdasarkan MAC address, dan juga mencegah konflik MAC di jaringan bridge jika banyak container berjalan bersamaan.

Sistem Metadata Container

Setiap container yang dibuat MELISA mendapatkan file metadata di /etc/melisa-info dalam format TOML. File ini menyimpan semua informasi yang diperlukan MELISA untuk mengelola container tanpa harus memanggil LXC setiap saat:

# /var/lib/lxc/[container]/rootfs/etc/melisa-info name = "myubuntu" distro = "ubuntu" release = "22.04" arch = "amd64" variant = "default" pkg_manager = "apt" slug = "ubu-22.04-x64"

Concurrent Distro Caching System

Mengambil daftar distribusi LXC yang tersedia adalah operasi jaringan yang lambat (bisa 10-30 detik). MELISA mengimplementasikan sistem cache concurrent yang canggih untuk mengatasi ini:

Konsep Jail Shell — Keahlian Tersembunyi

Konsep paling inovatif di MELISA adalah penggunaan MELISA itu sendiri sebagai login shell pengguna. Ketika admin membuat pengguna baru dengan melisa --add, perintah yang dieksekusi adalah:

Ini berarti ketika pengguna login ke server via SSH (ssh alice@server), shell yang diluncurkan adalah MELISA — bukan bash, bukan sh. Pengguna langsung masuk ke lingkungan MELISA, dan tidak ada cara untuk keluar ke bash host kecuali admin mengubah shell mereka.

SSH Pipeline & Remote Execution Engine

Remote execution engine MELISA menggunakan SSH bukan hanya sebagai saluran komunikasi — tetapi sebagai pipeline data yang sebenarnya. Pemahaman tentang ini adalah kunci untuk menghargai elegansi arsitektur klien MELISA.

Zero-Footprint Script Execution

Operasi melisa run adalah contoh terbaik. Ketika Anda menjalankan script Python lokal di container remote, tidak ada file yang pernah menyentuh filesystem server host:

Tar Streaming untuk Directory Transfer

# melisa upload mycontainer ./src /app tar -czf - -C ./src . | ssh server "melisa --upload mycontainer /app" # Sisi server (container.rs -- upload_to_container): # cat stdin | tar -xzf - -C /var/lib/lxc/mycontainer/rootfs/app # Seluruh direktori src/ ditransfer tanpa file temp!

Tanpa SSH multiplexing, setiap perintah MELISA dari klien akan memerlukan satu handshake SSL baru (200-500ms). Dengan ControlMaster yang dikonfigurasi oleh melisa auth add, semua koneksi berikutnya menggunakan socket yang sudah ada — latensi turun menjadi <20ms. Ini membuat MELISA terasa "lokal" meskipun semua eksekusi terjadi di server remote.

Lapisan Keamanan Berlapis 8 Tingkat

Model keamanan MELISA adalah implementasi defense-in-depth yang sangat komprehensif. Setiap lapisan dirancang untuk membatasi dampak jika lapisan sebelumnya berhasil dibobol. Ini bukan paranoia berlebihan — ini adalah desain yang tepat untuk sistem yang mengelola banyak pengguna di satu server.

1
Physical Handshake — Anti-Remote Initialization UNIK

Setup awal MELISA secara eksplisit memblokir eksekusi via SSH dengan mendeteksi keberadaan environment variables SSH_CLIENT, SSH_TTY, dan SSH_CONNECTION. Ini mencegah skenario serangan di mana penyerang yang berhasil mendapatkan akses SSH awal mencoba menginisialisasi sistem keamanan sebelum admin membangun pertahanannya. Tidak ada tool container lain yang menerapkan kontrol ini secara native. Setelah setup selesai, semua operasi bisa dilakukan via SSH — hanya inisialisasi awal yang diblokir.

2
SUID Binary + Jail Shell — Controlled Privilege Escalation UNIK

Binary MELISA dipasang dengan SUID bit (4755) di /usr/local/bin/melisa. Setiap pengguna yang dibuat melalui MELISA menggunakan MELISA sebagai login shell mereka. Kombinasi keduanya berarti: pengguna memiliki cukup privilege untuk mengelola container mereka sendiri, namun tidak pernah mendapat akses ke bash host secara langsung. Mereka "terkunci" di dalam lingkungan MELISA yang dikontrol.

3
Granular Per-User Sudoers (0440) — Least Privilege Enforcement ADVANCED

Setiap pengguna mendapat file sudoers tersendiri (/etc/sudoers.d/melisa_[user]) dengan permission strict 0440 (read-only oleh root dan group). Regular user hanya bisa menjalankan perintah LXC, Git, dan MELISA. Administrator mendapat izin tambahan untuk manajemen user dan system. Ketika pengguna dihapus, file sudoers mereka juga dihapus — tidak ada privilege yang tersisa.

4
Home Directory Isolation (chmod 711 + 700) — Anti-Enumeration ADVANCED

Direktori /home di-chmod ke 711 selama setup. Ini berarti pengguna biasa tidak bisa menjalankan ls /home untuk melihat daftar pengguna lain (execute tanpa read permission pada direktori). Ditambah dengan chmod 700 pada setiap folder pengguna individu, ini menciptakan isolasi lengkap antar pengguna — tidak ada yang bisa melihat atau mengakses file orang lain.

5
Immutable DNS Lock (chattr +i) — Anti-DNS Hijacking UNIK

Setelah container dibuat, file /etc/resolv.conf di dalam rootfs container dikunci dengan atribut immutable menggunakan chattr +i. Tidak ada proses di dalam container — termasuk root sekalipun — yang bisa memodifikasi atau menghapus file ini tanpa terlebih dahulu menghapus atribut immutable dari luar container. MELISA secara otomatis membuka kunci ini (chattr -i) saat container akan dihapus.

6
User Namespace Mapping (subUID/subGID 100000-165535) — Container Privilege Containment ADVANCED

MELISA mengkonfigurasi subordinate UID/GID mapping untuk setiap pengguna. Container berjalan dalam "unprivileged mode" — root (UID 0) di dalam container di-map ke UID 100000 di host. Jika terjadi container escape, penyerang hanya mendapat privileges user biasa (UID 100000) di host — bukan root. Ini adalah lapisan mitigasi container escape yang fundamental.

7
Orphaned Config Garbage Collection — Configuration Hygiene GOOD PRACTICE

MELISA secara aktif memindai dan membersihkan "orphaned" sudoers files — konfigurasi yang ditinggalkan oleh akun yang sudah dihapus secara manual. Sudoers orphan adalah vektor keamanan yang sering diabaikan: jika akun dengan nama yang sama dibuat ulang di masa depan, mereka bisa mendapat privilege dari konfigurasi lama yang seharusnya sudah tidak berlaku.

8
SSH Key-Based Auth + Ed25519 + ControlMaster — Secure Remote Channel ADVANCED

Klien MELISA menghasilkan keypair Ed25519 secara otomatis (jika belum ada) — bukan RSA 2048-bit yang lebih tua. Ed25519 menggunakan elliptic curve cryptography yang memberikan keamanan setara RSA 3072-bit namun dengan key yang jauh lebih kecil dan operasi yang jauh lebih cepat. SSH multiplexing memastikan sesi terautentikasi tetap hidup selama 10 menit, mengurangi frekuensi operasi kriptografi yang mahal.

🛡️ Mengapa Model Ini Disebut "Defense-in-Depth"?

Defense-in-depth berarti bahkan jika satu lapisan keamanan berhasil dibobol, lapisan-lapisan berikutnya masih membatasi kerusakan. Contoh skenario serangan dan bagaimana MELISA menghadapinya:

  • Skenario 1 — Remote setup attack: Penyerang mendapat akses SSH awal. Layer 1 (Physical Handshake) mencegah mereka menginisialisasi ulang sistem.
  • Skenario 2 — User enumeration: Penyerang login sebagai user reguler. Layer 4 (chmod 711 /home) mencegah mereka melihat pengguna lain.
  • Skenario 3 — Container escape: Penyerang berhasil keluar dari container. Layer 6 (user namespace mapping) membatasi mereka ke UID 100000 di host — bukan root.
  • Skenario 4 — DNS poisoning dari dalam container: Proses berbahaya di dalam container mencoba mengubah DNS. Layer 5 (chattr +i) memblokir modifikasi resolv.conf.

Desain Async & Mengapa MELISA Sangat Kompleks

Kompleksitas MELISA bukan berasal dari over-engineering, melainkan dari fakta bahwa ia menangani masalah yang memang inherently kompleks: orkestrasi sistem, keamanan multi-layer, kolaborasi tim, dan remote execution — semuanya dalam satu tool. Berikut adalah analisis mendalam setiap sumber kompleksitas:

Sumber Kompleksitas 1 — Full Async Architecture

Hampir setiap fungsi di MELISA menggunakan async/await. Ini bukan pilihan yang ringan di Rust — async Rust memerlukan pemahaman yang sangat mendalam tentang lifetimes, ownership, Pin, dan Send/Sync bounds. Namun ini diperlukan karena banyak operasi MELISA berjalan lama (pembuatan container bisa 10+ menit) dan tidak boleh memblokir thread utama:

// Sebuah fungsi yang terlihat sederhana, tapi menyembunyikan kompleksitas async yang dalam pub async fn create_container(name: &str, distro: &DistroMetadata) { ensure_host_network_ready().await; // async: network I/O run_lxc_create(name, distro).await; // async: subprocess + wait inject_network_config(name).await; // async: file I/O setup_container_dns(name).await; // async: file I/O + shell exec auto_initial_setup(name, pkg).await; // async: network + lxc-attach // Setiap .await adalah titik di mana Tokio bisa menjalankan task lain // Spinner animasi terus berjalan selama semua ini terjadi }

Sumber Kompleksitas 2 — Multi-Layer State Management

MELISA harus mengelola state di enam dimensi sekaligus tanpa database:

  • State LXC — container running/stopped (dari lxc-info)
  • State pengguna — /etc/passwd, /etc/sudoers.d/, /home/
  • State proyek — /opt/melisa/projects/ dan home directories
  • State jaringan — lxcbr0 bridge, firewall rules
  • State metadata — file TOML per container
  • State klien — ~/.config/melisa/profiles.conf, registry

Sumber Kompleksitas 3 — Dual-Mode Binary

Binary MELISA yang sama harus beroperasi dalam dua mode dari single entry point:

// main.rs — Routing dua mode dari satu binary if args.len() >= 2 { // MODE NON-INTERAKTIF: dipanggil via SSH (-c "command") // atau langsung dengan argumen (melisa --list) let cmd = if args[1] == "-c" { args[2..].join(" ") } else { args[1..].join(" ") }; execute_command(&cmd, &user, &home).await; exit(0); } else { // MODE INTERAKTIF: tampilkan banner, jalankan REPL display_melisa_banner(); melisa().await; }

Sumber Kompleksitas 4 — History Management yang Benar

MELISA mengimplementasikan manajemen history yang lebih canggih dari shell biasa. Alih-alih menulis ulang seluruh file history setiap perintah (seperti bash), MELISA menggunakan rl.append_history() yang hanya menambahkan baris baru — jauh lebih efisien. Error pun tetap dicatat agar user bisa menekan panah atas dan memperbaiki typo.

Sumber Kompleksitas 5 — Graceful Error Handling di Setiap Level

Setiap operasi async memiliki error handling yang berbeda: beberapa menggunakan eprintln!, beberapa return awal, beberapa menggunakan match dengan multiple arms. Ini diperlukan karena beberapa kegagalan (misalnya gagal update DNS lock) hanya perlu warning, sementara kegagalan lain (gagal stop container sebelum delete) harus mengabort seluruh operasi.

Spinner Async yang Benar

Modul loading.rs mengimplementasikan spinner progress bar yang berjalan secara konkuren dengan operasi async utama — sesuatu yang tidak trivial di Rust:

// loading.rs — Generic async spinner yang bisa dipakai oleh task apapun pub async fn execute_with_spinner<C, F, T>(message: &str, task_builder: C) -> T where C: FnOnce(ProgressBar) -> F, F: Future<Output = T>, { let pb = ProgressBar::new_spinner(); pb.enable_steady_tick(Duration::from_millis(80)); // 12.5 FPS // pb di-clone agar caller bisa memanggil pb.println() dari dalam task let result = task_builder(pb.clone()).await; pb.finish_and_clear(); result }

Ceruk Pasar yang Tidak Terlayani

MELISA memiliki posisi yang sangat spesifik namun nyata: ada ratusan ribu institusi pendidikan, lab penelitian, startup kecil, dan perusahaan yang membutuhkan lingkungan development server terisolasi dengan kontrol multi-user — namun tidak membutuhkan (dan tidak mampu mengelola) kompleksitas Docker Swarm atau Kubernetes. Untuk mereka, opsi saat ini sangat terbatas: SSH biasa tanpa isolasi, atau jump langsung ke Kubernetes yang butuh dedicated infrastructure engineer.

🎯 Positioning yang Tepat

  • Institusi pendidikan: Dosen bisa membuat container per mahasiswa untuk kelas pemrograman — satu perintah melisa --invite lab-week1 [semua NIM]
  • Startup tahap awal: Tim kecil berbagi satu server VPS dengan container terisolasi per developer
  • Lab penelitian: Eksperimen di container bersih tanpa kontaminasi lingkungan riset utama
  • Perusahaan SME: Environment development yang konsisten untuk tim tanpa butuh DevOps engineer dedicated

Peta Jalan Pengembangan — Dari Tool ke Platform

Berdasarkan analisis mendalam terhadap arsitektur yang ada, fondasi yang sudah dibangun pengembang sudah sangat solid untuk mendukung pengembangan dalam berbagai arah. Setiap item roadmap di bawah ini memiliki "pijakan" yang sudah ada di codebase saat ini.

Fase Saat Ini (v0.1.x) — Foundation Complete

Multi-Distro Support SUDAH ADA

Infrastruktur deteksi distro dan konfigurasi adaptif untuk Fedora/RHEL/Ubuntu/Debian/Arch sudah lengkap di host_distro.rs. Fondasi ini memungkinkan adopsi di hampir semua server Linux production.

Full Container CRUD + Lifecycle Management SUDAH ADA

Pembuatan, penjalanan, penghentian, dan penghapusan container sudah matang dengan safety checks di setiap transisi.

Multi-User dengan RBAC SUDAH ADA

Dua tingkat peran (Admin/Regular) dengan file sudoers per-user sudah berjalan.

Project Collaboration System SUDAH ADA

Invite, update, pull, out sudah berfungsi sebagai sistem kolaborasi dasar yang solid.

Remote Client via SSH Pipeline SUDAH ADA

Client Bash dengan zero-footprint remote execution, tar streaming, dan context detection sudah production-ready.

Fase Dekat (v0.2.x) — Enhancement Layer

Resource Quota per Container (CPU + RAM) SEGERA

LXC mendukung cgroups v1 dan v2 untuk membatasi CPU cores, RAM, dan disk I/O. Infrastruktur metadata TOML dan command executor sudah siap menerima parameter ini. Ini kritis untuk multi-user fairness.

Container Snapshot & Restore SEGERA

LXC mendukung lxc-snapshot secara native. MELISA bisa menambahkan --snapshot [container] dan --restore [container] [snapshot-id] sebagai perintah baru dengan minimal effort.

Template Container MELISA Custom SEGERA

Kemampuan untuk mendefinisikan "template" container dengan software pre-installed (misalnya template Python dev, template Node.js, template ML) yang bisa di-spin-up dalam hitungan detik tanpa konfigurasi ulang.

Perintah --logs [container] SEGERA

Menampilkan output dari syslog/journald container tanpa perlu masuk ke dalam container — berguna untuk monitoring cepat.

Fase Menengah (v0.3.x) — Platform Capabilities

Web Dashboard (REST API + Frontend) MASA DEPAN

Dengan arsitektur Tokio yang sudah ada, menambahkan HTTP server (axum atau actix-web) sangat natural. Dashboard web akan membuat MELISA accessible bagi pengguna non-CLI dan membuka pasar yang jauh lebih luas. Ini adalah transformasi MELISA dari "tool sysadmin" menjadi "platform developer".

Audit Log & Activity Monitoring MASA DEPAN

Modul tracing sudah ada di dependencies namun belum digunakan untuk persistent logging. Mengimplementasikan audit trail untuk semua operasi (siapa membuat container apa, kapan, dari IP mana) adalah syarat minimal untuk deployment enterprise.

Container Export/Import (Portable Bundles) MASA DEPAN

Mengekspor container sebagai tarball terkompresi dan mengimpornya di server lain — memungkinkan distribusi environment development yang konsisten antar mesin dan backup/restore yang mudah.

Network Policies per Container/Project MASA DEPAN

Saat ini semua container bisa berkomunikasi bebas via lxcbr0. MELISA bisa menambahkan kemampuan untuk mengisolasi container tertentu dari internet, atau membatasi komunikasi antar-container, menggunakan iptables rules yang di-generate otomatis.

Fase Lanjut (v1.x) — Enterprise Ready

Plugin System untuk Distribusi & Integrasi Eksternal MASA DEPAN

Arsitektur DistroMetadata dan DistroConfig yang sudah modular sangat cocok untuk sistem plugin. Ini memungkinkan penambahan container registry eksternal, notifikasi Slack/Discord, integrasi CI/CD, atau bahkan distribusi proprietary.

Multi-Node Federation (MELISA Cluster) MASA DEPAN

Kemampuan untuk mengelola banyak MELISA host dari satu panel — mendistribusikan container ke server yang memiliki resource tersedia. Ini adalah langkah menuju "mini-Kubernetes" yang jauh lebih simpel.

LDAP/Active Directory Integration MASA DEPAN

Integrasi dengan LDAP atau Active Directory untuk autentikasi terpusat — kritis untuk deployment di enterprise yang sudah memiliki infrastruktur identity management.

🔮 Potensi Diferensiasi Terbesar MELISA

Dengan penambahan Web Dashboard dan Resource Quotas, MELISA bisa menjadi alternatif self-hosted yang sangat kompetitif untuk platform seperti Coder, Gitpod, Okteto, atau DevZero — semua platform yang menyediakan "cloud development environment" — namun tanpa subscription fee dan tanpa ketergantungan pada cloud vendor. Ini adalah pasar yang sedang tumbuh sangat cepat dan MELISA memiliki fondasi teknikal yang tepat untuk memasuki ruang tersebut.
